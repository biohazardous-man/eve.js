EvEJS Ship Cosmetic Research
Updated: 2026-03-11

Scope
- Goal for now: unlock all ship skins for all characters and allow selecting an active skin for a ship.
- This note is for the practical implementation path, not a full SKINR clone.

Current conclusion
- Ship cosmetics are not a legacy dogma/inventory feature.
- There are at least two separate ship-cosmetic client paths in play:
  - `publicGatewaySvc` for newer `eve_public.*` ship cosmetic requests
  - `shipCosmeticsMgr` for older fitting/admin calls such as `GetEnabledCosmetics(shipID)`, `GetLicencedSkinsForShipType(charID, shipTypeID)`, `GetLicencedSkins()`, `GetFirstPartySkinData(licenseeID, skinID)`, `ApplySkinToShip(itemID, skinID)`, and `GiveSkin(skinID)`
- The V23.02 client uses `publicGatewaySvc` as a transport for newer `eve_public.*` protobuf/message-bus APIs.
- The old Macho `publicGateway` service stub is not the main cosmetic path for V23.02.
- The real cosmetic read path is the local HTTP/2 gRPC intercept behind `microservicesRedirectUrl`.
- A minimal unlock-all implementation still looks feasible without external microservices.
- The first-party fitting-window unlock path can be served locally through `shipCosmeticsMgr`.
- The repo now has a local gateway responder for ship skin state and the next step is in-game validation of the full fitting-window flow.
- `shipCosmeticsMgr` still matters for cosmetic fitting/rendering, but `GetEnabledCosmetics` itself can remain an empty dict in the first classic-skin phase.
- The current user-facing launcher/cert flow is being simplified so other users can reproduce the working ship-skin setup more easily.

  - input CA: `server/certs/xmpp-ca-cert.pem` + `server/certs/xmpp-ca-key.pem`
  - output leaf: `server/src/_secondary/express/certs/gateway-dev-cert.pem` + `gateway-dev-key.pem`
- `scripts/Install-EvEJSCerts.ps1` now handles the combined chat + public-gateway certificate setup for users.
- The gateway installer now:
  - trusts/distributes the CA certificate, not the gateway leaf
  - removes the earlier leaf-only trust hack from client PEM bundles when present
  - is idempotent on repeated runs
  - now emits the gateway server cert as `leaf + CA chain` in one PEM so stricter OpenSSL/BoringSSL clients can build the chain from the handshake alone
- Latest gateway diagnosis from the 2026-03-11 `skins3.txt` run:
  - the fitting window is successfully calling `shipCosmeticsMgr::ApplySkinToShip(shipID, skinID)`
  - the original local HTTPS proxy was receiving TCP connects for `dev-public-gateway.evetech.net:443`
  - but TLS reset before any `secureConnection` / HTTP2 stream was established
  - the first gateway installer built a self-signed server leaf and appended that leaf into client CA bundles
  - that is weaker than the chat setup, so the installer was changed to mint a gateway leaf signed by the existing repo CA (`server/certs/xmpp-ca-cert.pem`) and to trust/distribute the CA instead of the leaf
  - after that change, a second root cause was found:
    - the currently used server launcher was not setting `EVEJS_PROXY_LOCAL_INTERCEPT`
    - the live server process was only listening on `26000`, not `26003`
    - that meant the local HTTPS gateway intercept was not active at all in the latest test run
  - fixes now in source:
    - `server/src/_secondary/express/server.js` now defaults local intercept to enabled when `microservicesRedirectUrl` points at loopback
    - Windows wrappers were aligned with that path:
      - `scripts/windows/StartServerOnly.bat` now starts the server only and exports `EVEJS_PROXY_LOCAL_INTERCEPT=1`
      - `scripts/windows/RunClientProxy.bat` and `RunClientProxyAndDebug.bat` now use one shared config file and shared CA bundle settings
      - `scripts/windows/InstallCerts.bat` now gives users one explicit one-off cert install step instead of hidden installer chaining during launch
    - the current gRPC/TLS error changed from `invalid character on request` to explicit `CERTIFICATE_VERIFY_FAILED`
    - that is progress: the client is now reaching the local TLS endpoint, but its gRPC stack still does not fully trust the chain
    - latest follow-up fixes in source:
      - the generated gateway PEM now includes the CA chain
      - the gateway leaf profile was aligned with the known-good XMPP leaf profile:
        - no authority key identifier extension
        - basic constraints marked critical
        - key usage marked critical
        - extended key usage marked critical
      - Windows client launchers now set `SSL_CERT_FILE`, `GRPC_DEFAULT_SSL_ROOTS_FILE_PATH`, and `REQUESTS_CA_BUNDLE` to `server/certs/xmpp-ca-cert.pem`
    - local control result after that rebuild:
      - a temporary Node TLS server using `gateway-dev-cert.pem` + `gateway-dev-key.pem` now verifies successfully against `server/certs/xmpp-ca-cert.pem`
      - that means the on-disk gateway cert chain is now OpenSSL-compatible
      - remaining requirement: restart the real server so it presents the rebuilt gateway cert
    - next live-run issue after that:
      - the client reached `ALPN=h2` and the local secure connection completed
      - but the server crashed in `server/src/_secondary/express/server.js` because the HTTP/2 compatibility `request` handler still existed alongside the real `stream` handler
      - after ignoring the compat request once, Node still emitted an `IncomingMessage` error from the compatibility layer
      - fix now in source:
        - remove the secure-server compatibility `request` handler entirely
        - let the dedicated `stream` handler own all HTTP/2 gRPC traffic

Why cosmetics currently fail
- The client is not asking the old EVE service layer for ship skins.
- It is asking the gateway/message-bus layer for typed protobuf responses.
- Earlier runs failed because the server either returned dummy gateway data or never completed a trusted TLS handshake to the local HTTPS intercept.
- Result:
  - ship current-skin lookup fails
  - structure cosmetic fetches time out or fail
  - other modern systems on the same gateway also fail or time out
- There are currently two observed failure modes:
  - no working redirect/backend: requests time out and the client falls back silently
  - proxy/redirect path enabled but incompatible: requests fail fast with `500 invalid character on request`
- Additional root-cause correction from the latest pass:
  - `server/src/_secondary/express/certs/cert.pem` was a separate self-signed leaf cert that was not installed into the client trust chain
  - the local proxy tunnel reached `127.0.0.1:26003` and then reset during TLS, which prevented any real HTTP/2 gRPC stream from being established
  - a new scripted gateway-cert path exists, but the first version used a self-signed leaf rather than a CA-signed leaf
  - that fix is now implemented in source: the gateway certificate is generated from the existing repo CA and the installer distributes the CA instead of the leaf
