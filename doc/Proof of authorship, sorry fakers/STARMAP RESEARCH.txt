Where generic empty fallbacks are not enough
- `structureDirectory.GetJumpBridgesWithMyAccess`
  - `[]` is invalid
  - `( [], [] )` is also invalid
  - recovered compatible empty contract is:
    - `( [], [], [] )`
  - when populated, the first slot must contain pairs of endpoint objects with:
    - `solarSystemID`
    - `structureID`
- `insurgencySolarsystem.GetLocalCampaignClientSnapshot`
  - older runs returned `None`
  - later runs returned `[]`
  - decompiled `insurgencyCampaignSvc.py` now confirms the exact contract:
    - `(solarsystemID, campaignSnapshotOrNone)`
  - `client/undock2.txt` shows:
    - `retval=[]`
    - then `ValueError: need more than 0 values to unpack`
    - thrown from `insurgencyCampaignSvc.GetLocalCampaignSnapshot`
  - current source has now been patched to return:
    - `(currentSolarSystemID, None)`
- Practical conclusion:
  - generic empty-list fallbacks are not universally safe
  - some modern client paths are unpacking fixed tuples or expecting structured rowset/object payloads

Answer to the solar-system question
- We probably do not need to add a brand-new list of solar systems just to make StarMap open.
- The repo already has a full `solarSystems` dataset and a `stargates` dataset.
- In the StarMap crash logs, the client loads StarMap visual assets locally before the fatal service response arrives.
- The first observed server dependency during StarMap open is the jump-bridge overlay call, not a full "give me all solar systems" request.
- What we are more likely to need is:
  - correct jump-bridge link data
  - correct structure overlay data
  - correct insurgency / FW overlay contracts
- In other words:
  - the problem is more about service return shape and dynamic overlay content
  - not about missing the base map list on disk
