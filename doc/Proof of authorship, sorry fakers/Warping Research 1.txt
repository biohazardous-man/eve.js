2026-03-12

Warp research latest

Verified client-side facts

1. Warp effect loop facts from `_local/_tmp_Warp_decompiled.py`
   - `WarpLoop()` starts the warp indication immediately when `effects.Warping` begins.
   - The "active warp" branch is gated by `ball.effectStamp > 0`.
   - Tunnel progress is driven from the client's own `myPosition -> destination` distance once the effect is active.
   - `OnWarpStarted2` is not keyed off raw ball velocity. It is keyed off `shipModel.speed.value`.
   - Camera shake is not keyed off `shipModel.speed.value`. It is keyed off raw ball velocity relative to `ball.maxVelocity`:
     - `speed = (|ball.v| - ball.maxVelocity) / (ballpark.warpSpeed * AU - ball.maxVelocity)`
     - if `ball.maxVelocity` is pushed too high too early, shake headroom disappears.

2. Native destiny facts from `_destiny.dll`
   - Native `WarpTo` writes the warp destination, stop distance, warp factor, and warp-distance fields directly into the ball.
   - The native warp solver uses the same Rubicon-style exponential accel/decel math already matched by the server logs.
   - The remaining failure is therefore not the server curve. It is the pilot-side packet/state sequencing that decides when the client enters:
     - visible tunnel/effect
     - camera shake
     - sustained local warp progression

3. Cross-log facts that mattered
   - `warplogs57.txt`
     - pilot-only preactivation `SetMaxSpeed + WarpTo` produced real local motion before the official warp-start FX:
       - `20.47 -> 19.05 -> 16.96 -> 14.37 -> 11.42 AU`
     - once the delayed warp FX landed, the distance pinned at `11.42 AU` until the final landing anchor.
   - `warplogs58.txt`
     - adding one cruise `SetMaxSpeed` bump helped, but only moved the fixed-budget pin later:
       - `23.54 -> 22.05 -> 19.87 -> 17.17 -> 14.07 -> 10.59 -> 6.73 AU`
     - this proved the client still needed a cruise handoff, but it did not fix the missing shake or late tunnel.
   - `Warplogs60.txt`
     - same core failure on the current branch:
       - local motion before tunnel: `29.46 -> 27.97 -> 25.78 -> 23.08 -> 19.99 AU`
       - warp FX/tunnel only arrives later on the authoritative start stamp
       - distance pins at `19.99 AU`
       - final landing is still a hard anchor/stop snap
   - older research/logs (`warplogs4`, `warplogs6`, `warplogs19`, `warplogs21`)
     - the only branches that ever produced reported camera shake were the ones that injected pilot self `SetBallVelocity`
     - repeated or badly timed self velocity rebases caused the known regressions:
       - history rebases every tick
       - snapback
       - orientation fighting
       - ignored landing events

4. Root-cause map
   - The pilot preactivation hack was solving the wrong problem.
   - It let the pilot consume a chunk of local warp distance before the effect/tunnel started.
   - By the time `effects.Warping` arrived, the client was already partway through its local motion budget, so the tunnel appeared late and the remaining distance pinned.
   - The current branch also removed all pilot self warp velocity updates, which explains why camera shake disappeared again:
     - no raw velocity spike above `ball.maxVelocity`
     - high warp-time `SetMaxSpeed` seed left almost no shake headroom even if a velocity spike had arrived

5. Replacement model now implemented in `server/src/space/runtime.js`
   - Removed pilot preactivation from warp request time.
   - Restored the official warp-start sequence for the pilot:
     - seed `SetMaxSpeed`
     - `WarpTo`
     - immediate `effects.Warping`
     - `SetBallMassive(false)`
   - Lowered the pilot warp-speed seed back into a shake-safe range:
     - min `0.6`
     - max `0.7`
     - ceiling `0.7`
   - Restored exactly one pilot cruise `SetMaxSpeed` bump, but only when the warp actually has a real cruise leg.
   - Kept pilot-side warp `SetState` disabled at warp start.
   - Kept pilot-side in-warp `SetBallPosition` spam disabled.
   - Removed pilot startup-guidance `SetBallVelocity` again after `warplogs61.txt`.

6. What the next log should prove
   - no pilot preactivation `WarpTo` before official warp start
   - `WarpTo` and `OnSpecialFX effects.Warping` land together on the official warp-start stamp
   - no pilot self `SetBallVelocity` appears before landing
   - one later pilot cruise `SetMaxSpeed` appears once if the warp has a cruise leg
   - tunnel starts at entry instead of after most local motion is already spent
   - any remaining failure is isolated to the pure `seed + WarpTo + immediate FX + cruise bump` path instead of being masked by pilot self velocity rebases

8. The late-accel self velocity nudge is not just harmful, it is mathematically incapable of restoring shake on the current model.
   - In `warplogs61`, the late-accel nudge velocity magnitude was about `0.556 AU/s`.
   - The pilot seed `SetMaxSpeed` on the same run had already raised `ball.maxVelocity` to `3.5 AU/s`.
   - The decompiled client shake path computes:
     - `( |ball.v| - ball.maxVelocity ) / ( ballpark.warpSpeed * AU - ball.maxVelocity )`
   - That means the `1773287884` guidance packet had negative shake headroom by definition:
     - `0.556 AU/s - 3.5 AU/s < 0`
   - So the packet could only do one thing on this build:
     - rebase the pilot history
     - it could not wake shake

9. Replacement rule after `warplogs61.txt`
   - remove pilot startup-guidance `SetBallVelocity` entirely again
   - keep the simpler pilot path:
     - one warp-start seed `SetMaxSpeed`
     - one warp-start `WarpTo`
     - immediate warp effect
     - optional cruise `SetMaxSpeed` bump only
   - treat any future pilot self `SetBallVelocity` as invalid unless it exceeds the current seed `SetMaxSpeed` with enough margin to satisfy the actual client shake formula without forcing another history rebase

2026-03-12 native activation correction after `warplogs62.txt`

15. The warp-start pilot `SetMaxSpeed` was poisoning native activation.
   - `_destiny.dll` function `0x180036a20` contains the actual warp activation gate:
     - `mode == 3`
     - `+0x88 < 0` means still in the pre-activation align path
     - immediate activation requires:
       - heading error under `0.01` (`1 - dot(velocityDir, gotoDir) < 0.01`)
       - current speed squared at least `0.5625 * maxVelocity^2`
       - i.e. current speed `>= 0.75 * maxVelocity`
   - If that test fails, the client decrements `+0x88` each update and only forces warp once `abs(+0x88) > 180`, logging:
     - `AlignForWarp: Forcing warp as %I64d has been aligning for %d ticks.`
     - then `OnActivatingWarp`
   - This matches the live logs far better than the earlier `effectStamp` theory alone:
     - `warplogs60`: warp-start `SetMaxSpeed` at `1773286525`, `WarpState -> 2` only around `1773286531`
     - `warplogs62`: warp-start `SetMaxSpeed` at `1773288373`, `WarpState -> 2` only around `1773288381`
   - A 5-8 second delay is exactly what the forced-warp fallback would look like at the client's internal update rate.

16. Why the old pilot seed could never pass that gate.
   - `warplogs62.txt` shows the client still had valid pre-warp align state before warp start:
     - `1773288367`: `GotoDirection`
     - `1773288367`: `SetSpeedFraction(1)`
   - But on `1773288373` the pilot also received:
     - warp-start `SetMaxSpeed(485879988689.4973)`
     - warp-start `WarpTo`
   - That means the native activation test was comparing current subwarp speed against `0.75 * 3.25 AU/s`, which is impossible on the same tick.
   - So the pilot was guaranteed to miss immediate `OnActivatingWarp` and fall into the forced-warp path even before any later cruise bump arrived.

17. Corrected server rule now implemented.
   - Do not send a pilot warp-speed `SetMaxSpeed` on the same stamp as `WarpTo`.
   - Keep the pilot warp-start batch to:
     - `WarpTo`
     - immediate `effects.Warping`
     - `SetBallMassive(false)` on the same official start stamp
   - Keep the later pilot cruise `SetMaxSpeed` bump for long-profile warps only.
   - This preserves the native activation preconditions while still giving the pilot a later warp-speed ceiling once the client has actually entered warp.

2026-03-12 native WARP SetState decode correction after `warplogs63.txt`

18. The mode-3 `SetState` tail we had been sending was wrong.
   - `_destiny.dll` parser stub `0x1800928f0` feeds the richer warp initializer `0x180043140`.
   - That initializer does two extra things the plain `WarpTo` path does not:
     - seeds the ball's internal warp-start tick at `currentTick - 5`
     - seeds the ball's in-warp velocity vector at `+0x178/+0x180/+0x188`
   - The important detail is the call it makes into `0x1800425e0`:
     - it passes the three `goto` coordinates
     - it hardcodes stop distance to `0.0`
     - it passes only the warp factor as an integer
   - So the native mode-3 `SetState` tail is not:
     - `goto + effectStamp + warpDistance + stopDistance + warpFactor`
   - It is instead:
     - `goto(24) + warpFactor(4)`

19. Why that matters for the broken pilot path.
   - The old server WARP `SetState` encoder was appending:
     - `effectStamp`
     - `warpDistance`
     - `stopDistance`
     - `warpFactor`
   - That means the client was almost certainly decoding the wrong integer/double fields into the richer native initializer.
   - This cleanly explains why the earlier pilot `SetState` experiments changed controller state and local motion, but still never produced a stable active warp.

20. Corrected implementation now in the server.
   - `server/src/space/destiny.js`
     - WARP-mode `SetState` tail now encodes only:
       - stop-adjusted `gotoX/Y/Z`
       - `warpFactor` as an `int32`
   - `server/src/space/runtime.js`
     - the piloting client now gets the corrected WARP `SetState` at warp start instead of another pilot-side plain `WarpTo`
     - watchers still receive the normal `WarpTo` command path
     - pilot still gets immediate `effects.Warping`
     - pilot still gets `SetBallMassive(false)`
     - the bad same-stamp pilot warp-speed `SetMaxSpeed` remains disabled

21. What the next log should prove.
   - The pilot should no longer rely on the plain `WarpTo` initializer alone.
   - If the corrected native mode-3 `SetState` path is the missing piece, the next warp should show:
     - real tunnel progression during travel
     - distance continuing past the old fixed-budget pin
     - no need for pilot self `SetBallVelocity`
   - If it still fails, the next reverse-engineering target is no longer packet order:
     - it is the exact mode-3 `SetState` common-ball field set that `0x180043140` expects around the corrected `goto + warpFactor` tail.
