Warp Fix Notes 2026 Final

Date
- 2026-03-12

Root cause

1. The server's Rubicon warp curve was not the active problem.
- `docs/warpofficial.txt` and the server warp diagnostics already matched.
- The failure was on the piloting client's local warp-state handoff.

2. The current preactivation branch was poisoning the visible warp sequence.
- It sent pilot `SetMaxSpeed + WarpTo` at request/align time.
- That started local distance burn before the official warp-start FX.
- By the time the tunnel/effect began, the client had already spent a chunk of its own local warp budget and then pinned on the remainder.

3. Camera shake and tunnel progression depend on different client-side inputs.
- Shake depends on raw ball velocity relative to `ball.maxVelocity`.
- Full visible warp progression depends on the local warp-speed path (`shipModel.speed.value`) plus the effect/tunnel lifecycle.
- Raising pilot `SetMaxSpeed` too high too early removes shake headroom.
- Removing all pilot self `SetBallVelocity` updates removes the only input that ever previously woke the shake path.

Exact fix implemented

1. Removed pilot preactivation from warp request time.
- No more pilot-only `WarpTo` or pilot-only warp seed during align/request.
- Warp entry is now driven from the official warp-start tick again.

2. Restored the pilot's warp-start FX to the official start stamp.
- Pilot start order is now:
  - seed `SetMaxSpeed`
  - `WarpTo`
  - immediate `effects.Warping`
  - `SetBallMassive(false)`
- This is intended to put the tunnel at warp entry instead of after the local motion budget is already half spent.

3. Lowered the pilot warp-speed seed to restore shake headroom.
- seed scale floor: `0.6`
- seed scale default/ceiling: `0.7`
- this avoids the earlier `0.85-0.9` range that left almost no room for shake.

4. Restored one controlled cruise ceiling handoff.
- exactly one pilot cruise `SetMaxSpeed`
- only when the warp actually has a real cruise leg
- no repeated pilot max-speed rebases

5. Kept the known-bad paths disabled.
- no pilot warp-start `SetState`
- no pilot in-warp `SetBallPosition`
- no pilot startup-guidance `SetBallVelocity`
- no pilot preactivation warp branch

Files changed

- `server/src/space/runtime.js`

Why this fix is different

- It stops mixing two incompatible models:
  - "fake early local warp before FX"
  - "official start-time tunnel and cruise handoff"
- It removes the one pilot self velocity path that `warplogs61.txt` proved could only rebase history on a 1 Hz destiny schedule.
- It leaves the pilot on the simplest non-poisoning path still supported by the current evidence:
  - lower max-speed seed for local warp entry
  - one immediate warp effect
  - one later cruise max-speed handoff for sustained warp progression

What to verify in the next warp log

1. There should be no pilot `WarpTo` before the official warp-start stamp.
2. `Action: WarpTo` and `Action: OnSpecialFX ... effects.Warping` should occur together at warp start.
3. No pilot `Action: SetBallVeloc` should appear before the landing batch.
4. A single pilot cruise `Action: SetMaxSpeed` should appear later if the warp is long enough.
5. Tunnel should begin at entry instead of after most of the trip is already over.
6. The flat pinned remainder should move materially later or disappear.
7. If shake still does not return, the next thing to inspect is the client's local speed/model path, not another pilot self velocity nudge.

2026-03-12 activation-gate correction

New native root cause

1. The pilot warp-start seed `SetMaxSpeed` was structurally wrong.
- `_destiny.dll` warp activation (`0x180036a20`) requires the current ball speed to be at least `75%` of the current `maxVelocity` before it calls `OnActivatingWarp`.
- If that check fails, the client stays in the pre-activation align path and only forces warp after more than `180` ticks.
- The same function also logs:
  - `AlignForWarp: Forcing warp as %I64d has been aligning for %d ticks.`

2. Our previous pilot model guaranteed that activation check would fail.
- `warplogs62.txt` shows the pilot was already aligned and at `SetSpeedFraction(1)` before warp start.
- But on the warp-start tick we also sent a warp-speed `SetMaxSpeed`.
- That instantly raised native `maxVelocity` from subwarp speed to several AU/s before `WarpTo` activation was evaluated.
- So the client compared subwarp velocity against `0.75 * warp-speed maxVelocity`, which cannot pass.
- The observed late `WarpState -> 2` delays in `warplogs60/62` fit the forced-warp fallback, not a clean immediate activation.

Current fix

- Removed the pilot warp-start `SetMaxSpeed` override from the live warp-start batch in `server/src/space/runtime.js`.
- The pilot now gets:
  - warp-start `WarpTo`
  - immediate `effects.Warping`
  - `SetBallMassive(false)`
  - later cruise `SetMaxSpeed` only
- This keeps the pilot's native pre-activation gate on the real subwarp `maxVelocity`, which is the only state that can satisfy the `75%` activation test on entry.

2026-03-12 native WARP SetState decode correction after `warplogs63.txt`

18. The mode-3 `SetState` tail we had been sending was wrong.
- `_destiny.dll` parser stub `0x1800928f0` feeds the richer warp initializer `0x180043140`.
- That initializer does two extra things the plain `WarpTo` path does not:
  - seeds the ball's internal warp-start tick at `currentTick - 5`
  - seeds the ball's in-warp velocity vector at `+0x178/+0x180/+0x188`
- The important detail is the call it makes into `0x1800425e0`:
  - it passes the three `goto` coordinates
  - it hardcodes stop distance to `0.0`
  - it passes only the warp factor as an integer
- So the native mode-3 `SetState` tail is not:
  - `goto + effectStamp + warpDistance + stopDistance + warpFactor`
- It is instead:
  - `goto(24) + warpFactor(4)`

19. Why that matters for the broken pilot path.
- The old server WARP `SetState` encoder was appending:
  - `effectStamp`
  - `warpDistance`
  - `stopDistance`
  - `warpFactor`
- That means the client was almost certainly decoding the wrong integer/double fields into the richer native initializer.
- This cleanly explains why the earlier pilot `SetState` experiments changed controller state and local motion, but still never produced a stable active warp.

20. Corrected implementation now in the server.
- `server/src/space/destiny.js`
  - WARP-mode `SetState` tail now encodes only:
    - stop-adjusted `gotoX/Y/Z`
    - `warpFactor` as an `int32`
- `server/src/space/runtime.js`
  - the piloting client now gets the corrected WARP `SetState` at warp start instead of another pilot-side plain `WarpTo`
  - watchers still receive the normal `WarpTo` command path
  - pilot still gets immediate `effects.Warping`
  - pilot still gets `SetBallMassive(false)`
  - the bad same-stamp pilot warp-speed `SetMaxSpeed` remains disabled

21. What the next log should prove.
- The pilot should no longer rely on the plain `WarpTo` initializer alone.
- If the corrected native mode-3 `SetState` path is the missing piece, the next warp should show:
  - real tunnel progression during travel
  - distance continuing past the old fixed-budget pin
  - no need for pilot self `SetBallVelocity`
- If it still fails, the next reverse-engineering target is no longer packet order:
  - it is the exact mode-3 `SetState` common-ball field set that `0x180043140` expects around the corrected `goto + warpFactor` tail.

2026-03-12 protocol audit after the raw packaged-action suggestion

51. Framing audit after the "send each packet in its own frame" warning.
- The low-level network framing is already correct:
  - `clientSession.sendPacket()` writes one Macho packet as `[4-byte size][packet]`
  - the server does not concatenate Macho notifications as `[size][packet][packet]`
- But there was still a higher-level batching issue:
  - `sendDestinyUpdates()` groups same-stamp destiny actions into one `DoDestinyUpdate`
  - so the pilot warp-start path was still sending `WarpTo` and `OnSpecialFX('effects.Warping')` inside the same framed notification
- New correction:
  - keep the same stamp and the same verified command family
  - send the pilot warp-start actions as separate `DoDestinyUpdate` notifications (`splitUpdates`)
  - watcher updates remain unchanged
- Why this matters:
  - if the client performs reconciliation after each framed `DoDestinyUpdate`, the pilot can now process:
    - `WarpTo`
    - then `OnSpecialFX`
    - then `SetBallMassive`
    in distinct notification frames instead of one shared body
