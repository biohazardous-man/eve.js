EvEJS Space Research Notes
Updated: 2026-03-10

Scope
- These notes capture the packet, session, and movement rules used to finish the space-undock slice.
- Research used the local Crucible checkout and the extracted client artifacts as references only.
- No runtime code depends on those external references.

Primary conclusions

  - packet shapes were mirrored closely,
  - movement math was isolated and cleaned up instead of ported verbatim.

Session semantics confirmed
- Docked session state
  - `stationid` set
  - `stationid2` set
  - `locationid` set to station
  - `worldspaceid` set to station
  - `solarsystemid2` set
  - `solarsystemid` cleared
  - `shipid` set
- In-space session state
  - `stationid` cleared
  - `stationid2` cleared
  - `worldspaceid` cleared
  - `solarsystemid` set
  - `solarsystemid2` set
  - `locationid` set to solar system
  - `shipid` set

Dogma prime detail confirmed
- During an undock/session change, the client re-primes the active ship through `dogmaIM.GetAllInfo`.
- In that in-space prime, the ship `invItem.flagID` must remain `0`.
- Falling back from `0` to hangar-style `4` produces a logically invalid in-space ship row and can abort the undock after the animation starts.
- The same row must also carry the solar system ID as `locationID`.

Undock and bootstrap ordering confirmed
- Required order for stable client acceptance:
  - station guest removal notification
  - session mutation into space
  - dogma/session refresh
  - Beyonce bind
  - `GetFormations`
  - `AddBalls2`
  - `SetState`
  - immediate destiny follow-up updates
- The implementation follows that sequencing.

Destiny packet shapes mirrored
- `SetState`
  - includes `stamp`, `state`, `ego`, `damageState`, `aggressors`, `droneState`, `slims`, `solItem`, `effectStates`, `allianceBridges`
- `AddBalls2`
  - uses a binary state buffer plus extra ball data
- `DoDestinyUpdate`
  - carries stamped raw update payloads
- Removal/movement payloads mirrored
  - `RemoveBall`
  - `RemoveBalls`
  - `GotoDirection`
  - `SetSpeedFraction`
  - `Stop`
  - `SetBallVelocity`
  - `SetBallPosition`
  - `AlignTo`
  - `FollowBall`
  - `Orbit`
  - `WarpTo`
  - `OnSpecialFX`
  - `SetBallMassive`

Ball modes and binary sectors
- Implemented modes
  - `GOTO=0`
  - `FOLLOW=1`
  - `STOP=2`
  - `WARP=3`
  - `ORBIT=4`
  - `RIGID=11`
- Static objects
  - stations, stargates, and celestials are encoded as rigid balls
- Dynamic ships
  - include a common ball header
  - include mass and data sectors
  - include a mode-specific sector for the active destiny mode
- Mode-specific sectors now covered in the implementation
  - `STOP`
  - `GOTO`
  - `FOLLOW`
  - `WARP`
  - `ORBIT`

Movement and warp calculations used
- Useful Crucible-derived formulas retained as inputs
  - `massMKg = mass / 1_000_000`
  - `agility = massMKg * inertia`
  - `maxAccelTime = -ln(0.0001) * agility`
  - `alignTime = -ln(0.25) * agility`
- Implementation choices differing from Crucible
  - subwarp acceleration uses exponential blending toward a desired velocity
  - undock drift is produced from clean direction/speed state, not the rough Crucible hack
  - follow/orbit steering is resolved from explicit vector math and persisted target state
  - warp uses a generated accel/cruise/decel profile instead of a copied server tick hack

Cross-reference against Crucible behavior
- Kept
  - packet names and ordering
  - session-field semantics
  - ball mode numbering
  - `AddBalls2` plus `SetState` bootstrap pattern
  - rigid/static ball treatment for world objects
  - station dock metadata shape from `staStationTypes`
    - `dockEntryX/Y/Z`
    - `dockOrientationX/Y/Z`
  - undock startup shape
    - spawn at station `dockPosition`
    - apply undock ejection with heading from `dockOrientation`
    - use `SetBallVelocity` plus `GotoDirection` semantics instead of a stationary spawn
- Changed deliberately
  - no reliance on hardcoded station-only world data
  - no runtime dependency on external files
  - cleaned subwarp integration
  - explicit warp lifecycle handling
  - dynamic re-broadcasting of mode updates for late joiners
  - `solarsystem2` binding behavior for this client build
  - docked sessions keep `shipid` populated because this V23.02 client breaks station ship presentation and boarding updates if it is cleared

Database and data-shape findings
- Static world/runtime data must be available in the same workspace-local storage format as mutable gameplay state.
- The implemented solution stores the imported Fuzzwork/SDE-derived tables under `server/src/newDatabase/data`.
- Runtime services consume those tables through shared loaders, so the space code path does not split between SQL, raw CSV, or external file references.
- The remaining station-model mismatch was not solvable from `staStations.csv` alone.
- Crucible confirmed the missing data source: `staStationTypes`.
- Fuzzwork still publishes that file, and it contains the real dock-entry and dock-orientation transforms needed to line the runtime ballpark up with the rendered station model.

Verification findings
- The client-facing space slice accepted:
  - undock bootstrap,
  - movement updates,
  - multi-player visibility,
  - docking removal,
  - warp replication,
  - stargate jump re-entry.
- Two-character in-process smoke verification confirmed the expected sequence and successful cross-player propagation.
- A follow-up undock validation confirmed that `dogmaIM.GetAllInfo` now returns the active ship with `locationID=30000142` and `flagID=0` once the session is in space.
- V23.02 remote `SetState` unmarshal is sensitive to rowset class names during live space notifications. In the undock bootstrap path, `droneState` must serialize as `eve.common.script.sys.rowset.Rowset` instead of bare `util.Rowset`, or the client throws `module object has no attribute 'Rowset'` while processing `DoDestinyUpdate`.
- `godma.RefreshTargets()` immediately calls dogma remote `GetTargets()` and `GetTargeters()` after the ship becomes active. Returning `None` here causes the client-side `'NoneType' object is not iterable` failure during the first space session prime; returning empty lists is accepted.
- V23.02 `michelle.SetState` reads `industryLevel` directly from the `KeyVal` bag during the first real space flush. Omitting it leaves the client in a repeated exception loop even if the rest of ballpark bootstrap succeeds.
- After `industryLevel`, the same V23.02 `SetState` path also reads `researchLevel` directly from the same `KeyVal` bag. Leaving it out produces the next `KeyVal instance has no attribute 'researchLevel'` failure during `RealFlushState`.
- The same `michelle.SetState` path then reads `dbuffState`. An empty dict is accepted for the current emulator slice; omitting the field causes the remaining periodic jitter loop because each flush retries and faults again.
- Jita stargates must carry a populated `jumps` rowset in their slim items. Empty or missing jump rows lead to the client stargate controller failing during scene object preparation.
- The piloting client should not receive authoritative `SetBallPosition` or `SetBallVelocity` corrections every movement tick during normal subwarp flight. Sending those to watchers only, with periodic cadence, avoids self-rubberbanding while still keeping observers in sync.
- `bountyProxy` is not optional once the space scene is live. The client starts fetching bounties and kill-right data for loaded balls immediately after bootstrap, so returning no service there creates an avoidable secondary error flood.
- Persisted in-space `GOTO` state is unsafe to restore directly on session entry for this build. Reattaching a character already in space should normalize the active ship to `STOP`, zero velocity, and current-position target point.
- `beyonce.UpdateStateRequest` must not force a fresh initial ballpark once `initialStateSent` is already true. Replaying `AddBalls2` plus `SetState` on that probe causes visible scene churn, overview object flicker, and repeated static-object respawn behavior.
- Even with valid `jumps` rowsets, V23.02 still rejects the current stargate scene payload during `clientControllerRegistry.StargateSetup`. Until the missing controller field is identified, excluding stargates from the rendered scene is safer than letting them destabilize the whole ballpark.
- The V23.02 client also issues `eve_public` suppression / insurgency gRPC requests while space stations are loading. If those escape to CCP with emulator-local character IDs, the official service returns `500 invalid character on request`, which in turn causes repeated `_SetCorruptionAndSuppression` tasklet failures on station objects. The debug client must therefore launch with the local proxy enabled, and the local proxy must intercept those `evetech` / `eveonline` public-service hosts with harmless stub responses.
2026-03-10
- The previous docking path was still wrong in two ways:
  - `beyonce.CmdDock` docked immediately without checking distance.
  - `beyonce.CmdFollowBall(stationID, 50)` was being treated as an immediate dock shortcut instead of a docking-approach intent.
- For this V23.02 client, the safer rule is:
  - `CmdFollowBall(stationID, 50)` begins docking approach,
  - `CmdDock` also degrades into docking approach if the ship is still out of range,
  - the runtime completes docking only after the ship actually reaches station docking range.
- `charMgr.GetHomeStation()` was also shaped incorrectly for the client-side `homestation` flow. Returning a raw ID or `None` is not sufficient; the client expects an object with at least `.id`.
- Undocking exposed that bug more often because character records dropped current `stationID` to `null` without preserving a separate home-station field. The fix is to keep stable `homeStationID` / `cloneStationID` on the character record and answer home-station lookups from those fields instead of current docked state.
- Docking investigation: server traces showed the client was not sending `CmdDock` when the user pressed dock from overview; it repeatedly sent `beyonce.CmdFollowBall(stationID, 50)` instead. That means the immediate problem was not the dock handler itself, but the client/server shared idea of station distance and arrival.
- The imported Jita 4-4 station row had `radius=30000` and `undockPosition` about 32500 m from station center. The runtime was also using that same 30000 m radius for live station entities, so follow math computed a desired arrival range around `followRange + shipRadius + stationRadius`, i.e. roughly 30 km. This explains overview showing the station around 35 km away and dock clicks degrading into approach without ever closing properly.
- Fix direction: keep the large station radius in reference data for future rebuild provenance if needed, but use a small operational interaction radius in the live runtime and compute undock spawn from that smaller docking perimeter.
- Follow-up finding: after the geometry fix, existing characters could still log in at `~72-75 km` from Jita 4-4 center because that bad approach state had already been persisted into `items.data.json`. On this client, the large station model can still make that look visually adjacent even when the server-side ballpark position is far out. The runtime therefore needs to normalize legacy near-station stopped/follow states on attach, not just future undocks.
- Correction to the above fix direction: shrinking the live station ball to `1000 m` was not viable for this client. The overview and destiny distance logic then measured against a tiny shell while the rendered station model still used the real station art footprint, which produced the "ship inside structure at 6-7 km" and "double-click causes a visible teleport" reports.
- The safer rule is: live station entities should keep their real radius, while docking permissiveness is handled separately. Undock spawn and legacy near-station normalization should use a perimeter outside the real station radius, preferring the stored undock vector length when it is already outside the station mesh.
- Separate but related finding: `moveShipToSpace()` was still merging the ship's previous stored `spaceState` into a new undock/jump state. For docked ships that retained stale follow/goto data, this allowed old target-tracking fields to survive into the next undock even when position, velocity, and mode were reset. Fresh space transitions should overwrite the state instead of merging it.
- Additional persistence finding: `dockShipToStation()` was correctly writing `spaceState: null`, but `normalizeShipItem()` was using nullish fallback semantics and restoring the previous `spaceState` from defaults. That meant docked ships could remain flagged as `FOLLOW` with non-zero velocity in the persisted items table even while `locationID` and `flagID` said the ship was in station.
- Safe normalization rule: explicit `spaceState: null` must survive normalization, and any ship not in space (`flagID != 0`) should be treated as having no `spaceState` regardless of what old JSON rows still contain.
- Movement-authority finding: watcher-only subwarp corrections are not sufficient once the piloting client drifts from the server's idea of ship position. Double-click / direction-change commands then execute from different origins on client and server, which looks like overview distance teleportation and can delay docking completion by tens of seconds.
- Safer runtime rule: send authoritative position/velocity re-anchors to the piloting client at movement-command boundaries and include the piloting client in throttled periodic subwarp corrections. Full every-tick self-corrections are still too aggressive, but zero self-correction leaves the server/client positions diverged.
- Live debugging rule: a separate `space-movement-debug.log` is useful because packet call traces alone show the user actions (`CmdGotoDirection`, `CmdFollowBall`, `CmdDock`) but not the server-side ship coordinates or dock-distance math at the moment those requests are handled.
- Follow-up from live movement tracing: once docking intent is active, the server now shows a smooth monotonic approach to the dock point and reaches `dockReady` correctly at about `2490 m` docking distance. The remaining visible jitter is therefore not a fundamental inability to dock, but client presentation disturbance during a valid approach.
- Repeated client-side `CmdFollowBall(stationID, 50)` / `CmdDock` retries should be treated as idempotent while the same docking approach is already active. Re-clearing tracking state and rebroadcasting the same follow each time adds visible reorientation churn without improving convergence.
- A practical compromise for self-correction is asymmetric cadence:
  - observers can keep the tighter correction interval,
  - the piloting client should get a slower periodic correction stream plus immediate command-boundary anchors.
  This keeps the server/client positions aligned without reintroducing the half-second self-rubberband on every movement path.
- Separate from docking math, some client home-station flows appear to consult live session state while the character is already in space. Keeping `homeStationID` / `cloneStationID` only in the character record is not sufficient for those paths; the live session should carry those IDs too.

Server logs showed docking approach was being integrated toward `station.dockPosition`, while the client only had `FollowBall(stationID, 50)` plus the station ball radius. That means the client was predicting approach against the station ball, while the server was flying toward a different hidden point. Result: visible reorientation/reset jitter even though server-side convergence eventually finished.

That can only happen if docking distance is still being measured against a custom dock point instead of the station ball geometry the client knows about.
- Fix direction: use imported dock metadata for reference/debug only, not for live `FollowBall` steering or docking readiness. The authoritative movement path must match what the V23.02 client can infer from the ballpark: station center, station radius, follow range, and velocity corrections.
- The extra dock/login black-screen still does not show up as a second server session change in `session-change-debug.log`. The next strongest suspect remains client post-transition service/bootstrap behavior, especially the `home_station`/`GetHomeStation` path that has previously produced `data=None` client exceptions. Alias services and extra packet tracing were added for the next validation pass.

Returning only camelCase station fields is therefore still invalid for V23.02 even after the service alias itself exists and resolves correctly.

- The cleanest data model is to keep `homeStationID` / `cloneStationID` on the character record as the single authoritative source of truth, then derive session/service payloads from those fields. Current `stationID` is transient docked state and should not be trusted as home-station truth once the character is already in space.

2026-03-10
- Taking a step back against Crucible clarified the actual subwarp mismatch: the remaining snap did not happen during steady-state movement, it happened when the user issued a new movement command. Our runtime was rebroadcasting `SetBallPosition` and `SetBallVelocity` on every `GotoDirection`, `AlignTo`, `FollowBall`, `Orbit`, `SetSpeedFraction`, and `Stop`. Crucible's normal destiny command path does not do that for the piloting client, so the client and server were being forcibly re-anchored at exactly the command boundary where the user noticed the teleport.
- Another structural mistake was conflating approach with dock intent. Treating every `CmdFollowBall(stationID, 50)` as docking intent explains why ordinary approach could eventually dock and why docking state was being re-entered from generic follow traffic. Safe rule: only `CmdDock` may arm docking intent; `CmdFollowBall` is just movement.
- Crucible's dock flow is also staged: out-of-range `CmdDock` raises `DockingApproach`, in-range `CmdDock` starts a dock timer and returns a file time, `OnDockingAccepted` is sent during the accepted phase, and only then does the actual station session transfer happen. Immediate `dockSession()` at the moment server range reaches green is too early and is a strong match for the observed client-side double re-entry.
2026-03-10
- The next deeper movement mismatch was in the server integrator itself. Our previous `applyDesiredVelocity()` blended the full velocity vector toward the target vector using `alignTime / 3`, which reaches commanded speed far too quickly and also couples turning and acceleration into one shortcut. That is not how destiny movement is modeled in Crucible.
- Crucible keeps speed changes logarithmic and separate from heading changes. The practical reusable part for EvEJS is the scalar speed model: derive ship agility from `maxAccelerationTime / ln(10000)` (or `alignTime / ln(4)` as fallback), then move the current speed fraction toward the user-commanded speed fraction with an exponential curve. This matches the dataset already generated into `movementAttributes`.
- Undock was also structurally wrong: `buildUndockMovement()` was setting full subwarp velocity on attach. That means the server began several seconds ahead of the client immediately after undock, before any player input. Safe rule: undock should set direction + commanded speed fraction, but velocity should still start at zero and be produced by the normal tick/acceleration path.
- Repeating `SetSpeedFraction` on every `GotoDirection` / `FollowBall` / `Orbit` even when the fraction did not change is another avoidable desync source. The client already knows its throttle unless the server explicitly changes it; unnecessary speed-fraction echoes restart local motion state without adding authority.
2026-03-10
- Decompiled V23.02 `michelle.DoDestinyUpdate()` contains a critical rule we had still violated: every `DoDestinyUpdate` batch must contain only one timestamp. The client explicitly logs an error and treats mixed-timestamp batches as a desync condition:
  - it collects all timestamps in the batch,
  - if `len(timestamps) > 1`, it logs every action and calls `clientStatsSvc.OnFatalDesync()`.
- EvEJS was still emitting exactly that invalid shape in several runtime paths:
  - ship prime updates (`SetBallAgility`, `SetBallMass`, `SetMaxSpeed`, `SetBallMassive`),
  - command updates where a movement command and a `SetSpeedFraction` were bundled together with different stamps,
  - docking accept updates (`SetSpeedFraction`, `Stop`, `SetBallVelocity`),
  - generic correction updates (`SetBallPosition`, `SetBallVelocity`).
- Safe rule for this client build: either:
  - all actions in one `DoDestinyUpdate` share the same stamp, or
  - the server sends multiple `DoDestinyUpdate` notifications, one per stamp group.
- Crucible `DestinyStructs.h` also confirmed that our live `GOTO` ball sector was still malformed. `GOTO_Struct` is:
  - `uint8 formationID`
  - `double x`
  - `double y`
  - `double z`
  That `x/y/z` triple is the target point, not the normalized direction vector. Crucible `DestinyManager::GotoDirection()` sets `m_targetPoint = direction * 1.0e16`, which matches the client-side prediction model much better than our previous unit-vector encoding.
- The warp FX failure was also confirmed from the decompiled V23.02 client:
  - `fxSequencer.OnSpecialFX(self, shipID, moduleID, moduleTypeID, targetID, otherTypeID, area, guid, isOffensive, start, active, duration=-1, repeat=None, startTime=None, graphicInfo=None)`
  - our user log showed `guid=()` and `isOffensive='effects.Warping'`, which is only possible if the payload shape/order is wrong at runtime.
- Safe `OnSpecialFX` emission rule:
  - use the `OnSpecialFX14` shape,
  - send `area` as an actual list object, not a tuple-shaped nested array,
  - keep `guid` in slot 6,
  - include the optional duration/repeat/startTime/graphicInfo fields so the client's long-form handler is satisfied for warp startup.
- Static celestials should also keep their imported semantic kind (`sun`, `planet`, etc.) in the live runtime entity. Flattening everything to generic `celestial` hides meaningful behavior differences and was already breaking type-specific warp stop logic in EvEJS itself.
2026-03-10
- Crucible `DestinyManager::BeginMovement()` and `SetSpeedFraction(..., startMovement=true)` explain a subtle but important client expectation: direction/approach/orbit changes are not just heading changes, they also restart the client's movement state even when the commanded throttle fraction stays the same. Safe rule for EvEJS:
  - on `GotoDirection`, `AlignTo`, `FollowBall`, and `Orbit`, reissue `SetSpeedFraction(currentFraction)` alongside the command update,
  - do not require the numeric speed fraction to change before sending it.
- Crucible also strongly suggests that routine ship movement should not be corrected with repeated `SetBallPosition` packets. `SetPosition(..., update=true)` is used sparingly and even the reference comments frame position forcing as a debug / sync hack. Safe rule:
  - ordinary subwarp should be command-driven plus local destiny simulation,
  - `SetBallPosition` should be exceptional (attach / hard resync / warp),
  - if correction is still needed during subwarp, prefer `SetBallVelocity` first.
- The newest symptom change from live testing is diagnostically meaningful:
  - when the ship itself visibly snaps, the piloting client is being re-anchored too aggressively;
  - when the ship looks stable but the whole world jitters around it, the piloting client is free-running on local prediction while the rest of the world stays on server authority.
  This means the correct fix is not simply "more" or "less" correction, but getting the movement-start handshake and ongoing ship simulation closer to the real client expectation so hard corrections are mostly unnecessary.
2026-03-10
- The boarding/login dogma activation contract is build-sensitive, and the live client log must take precedence over a conflicting decompile snapshot. The client currently under test throws `ValueError: need more than 3 values to unpack` when given a 3-slot activation tuple, so the compatible live rule for this build is:
  - `ship.ActivateShip` returns a 4-slot activation tuple,
  - dogma `allInfo.shipState` also returns a 4-slot tuple,
  - the first three slots remain the usual instance-cache / charge-cache / weapon-bank data,
  - the fourth slot is currently treated as a reserved compatibility payload and can remain empty.
- Live client logs overrule the older decompiled `OnDockingAccepted` signature assumptions for this build. The active services (`svc.tacticalNavigation`, `svc.space`, `svc.dockingHeroNotification`) faulted with `OnDockingAccepted() takes exactly 2 arguments (4 given)` when EvEJS spread the dock-accepted payload into three notification args. Safe rule: send `OnDockingAccepted` as one notification argument containing the `[dockStartPos, dockEndPos, stationID]` payload object.
- Movement tracing identified a server-integrator bug independent of packet shape: after `CmdGotoDirection`, the next tick still reconstructed heading from the previous velocity vector if velocity was non-zero. That makes the server drift back toward the old heading even though `targetPoint` and the client command point elsewhere. Safe rule: commanded `direction` must outrank stale `velocity` when building desired heading.
- Current simulation harness now proves four core contracts before live restart:
  - boarding produces one `shipid` session change per activation,
  - undock produces one station-to-space session change,
  - repeated `GotoDirection` commands keep server heading and velocity aligned to the fresh command,
  - dock accept produces the corrected one-argument `OnDockingAccepted` notification shape and one final dock session change.
2026-03-10
- Decompile-first correction for normal subwarp commands:
  - The live V23.02 client's double-click navigation path in `eve/client/script/ui/inflight/navigation.py` sends only `CmdGotoDirection(pickDir.x, pickDir.y, pickDir.z)`. It does not automatically send `CmdSetSpeedFraction()` on each heading change.
  - That matters because EvEJS was still echoing `SetSpeedFraction(currentFraction)` back to the piloting client on every `GotoDirection`, `AlignTo`, `FollowBall`, and `Orbit`, even when the numeric throttle had not changed. Safe rule from the decompiled client command flow: only send `SetSpeedFraction` when movement is actually starting from rest or when the numeric throttle changes.
  - The server-side heading model was also structurally wrong. If `gotoDirection()` immediately overwrites the live heading, then the tick integrator no longer performs a real turn; it starts from the target heading and only changes speed. Safe rule: movement commands should update the commanded target point / commanded direction, while the persisted live heading should continue to represent the ship's current heading and rotate toward the command over time.
- `OnDockingAccepted` correction after live client evidence:
  - The previous "one outer payload object" rule was disproved by the newer client traceback: `spaceMgr.OnDockingAccepted -> GetWarpDestinationName -> cfg.evelocations.Get(key=((...), (...), stationID))`.
  - That traceback means the whole docking triple was arriving where `stationID` should have been, i.e. it was wrapped one level too deeply.
  - Safe rule for the currently tested build: `OnDockingAccepted` notification args should be spread as the normal 3-item tuple `[dockStartPos, dockEndPos, stationID]`, not wrapped in an extra outer tuple/list.
- New diagnostics:
  - `server/logs/space-destiny-debug.log` now records every outbound `DoDestinyUpdate` sent to a client, including stamp, destiny function name, and summarized args. This should make any remaining piloting-client resync immediately attributable to a concrete outgoing packet sequence rather than guesswork.
2026-03-10
- Fresh live client traceback settles the current V23.02 `OnDockingAccepted` contract again: `DockingHeroNotificationService.OnDockingAccepted() takes exactly 2 arguments (4 given)` with runtime args `((...), (...), stationID)` means EvEJS must send one notification argument whose inner payload is `[dockStartPos, dockEndPos, stationID]`. Safe rule for this tested build: `session.sendNotification("OnDockingAccepted", "charid", [[dockStartPos, dockEndPos, stationID]])`.
2026-03-10
- Warp follow-up from fresh live testing:
  - The running V23.02 client is still happier with the short warp startup effect shape used by Crucible, but the live traceback tightened it further: this tested client path behaves as if there is no positional `area` argument in the callback. The signature symptom `guid=[]` and `isOffensive='effects.Warping'` means EvEJS inserted one extra list slot before `guid`. Safe rule for current warp startup: send `OnSpecialFX` as `[shipID, moduleID, moduleTypeID, targetID, otherTypeID, guid, isOffensive, start, active]` with no `area` slot.
  - Immediate `mode='WARP'` on command is too early. Safer rule: warp command first enters a normal align/accelerate phase using subwarp movement, then sends `WarpTo` once heading is within about `6 degrees` and actual speed is at least `~75%` of max.
